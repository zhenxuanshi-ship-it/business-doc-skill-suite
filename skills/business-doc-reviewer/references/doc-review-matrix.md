# Document Review Matrix

## feasibility-report

Primary review focus:
- necessity is real or inflated
- options comparison is meaningful
- recommendation is evidence-based
- feasibility dimensions are complete
- risk section is substantive

Typical killer problems:
- fake option comparison
- recommendation already decided before analysis
- business benefits are slogans
- cost and complexity are hidden

## project-proposal

Primary review focus:
- approval logic is clear
- objectives and scope are specific
- expected outcomes are plausible
- budget/resource request has basis
- implementation path is believable

Typical killer problems:
- asks for approval without a clear ask
- outcomes are not measurable
- scope too broad for resources

## prd

Primary review focus:
- user and scenario clarity
- scope/non-scope separation
- requirement testability
- acceptance criteria completeness
- missing non-functional requirements

Typical killer problems:
- requirement statements too vague to build
- no acceptance criteria
- product goal and feature list do not match

## architecture-design

Primary review focus:
- module boundary clarity
- technical trade-offs are explicit
- data flow and integration flow make sense
- security/performance/reliability are real
- deployment path is credible

Typical killer problems:
- technology choices without reasons
- architecture diagram implied but not explained
- no trade-offs, only ideal-state claims

## implementation-plan

Primary review focus:
- milestone realism
- responsibility clarity
- dependency visibility
- execution sequencing
- delivery risk and mitigation

Typical killer problems:
- timeline is optimistic fiction
- owners are unclear
- key dependencies are hidden
