# PRD Guide

## Default structure

1. Background and goals
2. Target users and scenarios
3. Scope and non-scope
4. Functional requirements
5. Non-functional requirements
6. Dependencies and constraints
7. Milestones
8. Acceptance criteria
9. Risks

## Requirement writing rules

- one requirement, one meaning
- avoid mixing solution detail into user-level requirement unless needed
- each important requirement should be testable
- define non-scope explicitly
- acceptance criteria should be observable

## Useful requirement fields

- ID
- title
- priority
- description
- rationale
- acceptance criteria
- dependency

## Common PRD problems

- goals are slogans, not product outcomes
- scenarios are vague
- scope and non-scope are mixed together
- acceptance criteria are missing
- non-functional requirements are forgotten
